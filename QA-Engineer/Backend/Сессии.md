1. **Что такое сессия в веб-приложении, и как она используется?**
   - Ожидаемый ответ: объяснение, что сессия используется для сохранения состояния пользователя между HTTP-запросами, так как HTTP — это статeless-протокол.

2. **Как происходит создание и хранение сессии на сервере?**
   - Ожидаемый ответ: создание сессии, обычно после аутентификации, хранение информации на сервере в виде уникального идентификатора, связанного с данными пользователя.

3. **Какие есть методы передачи session ID между клиентом и сервером?**
   - Ожидаемый ответ: через cookies, URL-параметры, HTTP-заголовки.

4. **Каковы риски хранения session ID в cookies и как их можно минимизировать?**
   - Ожидаемый ответ: риски, такие как XSS (Cross-Site Scripting) и CSRF (Cross-Site Request Forgery), и методы их уменьшения, например, использование атрибутов HttpOnly, Secure, SameSite.

5. **Что произойдет, если session ID будет утерян или скомпрометирован? Каковы последствия и как это предотвратить?**
   - Ожидаемый ответ: возможен захват сессии (Session Hijacking), обсуждение механизмов предотвращения — например, использование двухфакторной аутентификации, регенерация session ID после входа.

6. **Как сессии истекают (expire)? Какие подходы к управлению временем жизни сессии существуют?**
   - Ожидаемый ответ: можно настроить таймауты для неактивных сессий или установить жесткое время жизни сессии. Важно поддерживать баланс между безопасностью и удобством для пользователя.

7. **Что такое sticky sessions (сессии с привязкой) и какие их плюсы и минусы в контексте масштабируемости backend-систем?**
   - Ожидаемый ответ: это сессии, которые сохраняются на одном сервере в случае использования нескольких серверов (кластеров). Преимущества — уменьшение нагрузки на серверы хранения сессий, недостатки — сложности с балансировкой нагрузки и возможные потери данных при сбое сервера.

8. **Какие способы масштабирования системы с сессиями существуют, когда приложение обслуживается несколькими серверами?**
   - Ожидаемый ответ: использование распределенного хранилища сессий (Redis, Memcached), sticky sessions, JWT (JSON Web Tokens) как альтернативы сессиям.

9. **Чем отличается хранение состояния через сессии и через JWT?**
   - Ожидаемый ответ: JWT не требует хранения на сервере и используется как самодостаточный токен, который содержит информацию о пользователе. Сессии же требуют серверной памяти, где хранится состояние, что добавляет нагрузку на сервер.

10. **Как тестировать корректную работу сессий в веб-приложениях? Какие кейсы нужно проверить?**
    - Ожидаемый ответ: проверка работы сессий при разных пользователях, проверка истечения сессий, атаки на захват сессий (например, при повторном использовании session ID), правильное завершение сессии (logout).
